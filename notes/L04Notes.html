Testing and Debugging

 

·      Bugs are impossible to avoid and we need to know how to handle them

·      There are three types of errors:

o   System error – conflict with the environment or hardware that is used

o   Programmer error – faulty code

o   User error – the user put in data that the code can’t handle

·      Exception – returns the value of the error and a stack trace

·      Stack Trace – the exact location of where the code stopped running

·      Warning – an error that the code managed to work through but should still be fixed

·      Strict Mode – throws warnings on working but sloppy coding practices

·      To use strict mode place ‘use strict’; at the first line of your .js file

·      If you’re adding to someone else’s working code, place all your code in a self-invoking function with strict on like this:

 

(function() {

     ‘use strict’;

     // code here

}());

 

·      this way, it won’t apply ‘strict’ on any working code you didn’t write.

·      Linting tools help prevent sloppy and non-functional code from being produced

·      Feature Detection is a way of not running certain code if the browser doesn’t support it

·      Here is an example of feature detection:

 

if (window.yourAPI) {

     // call code here

}

 

·      alert() – classic debugging tactic

·      you can also debug through log statements on the console

·      console.trace() – returns a stack trace to that line. Useful for debugging

·      You can also set break points and debug in the browser tools

·      const error = new Error(); – allows you to anticipate an exception

 

 

 

 

 

JavaScript errors:

 

·      Each error has the properties .name, .message, .stack

·      throw x causes the program to stop and raises x

·      It is a good practice to only throw Errors

·      try { } catch(error) { } finally { }; – allows you to write custom code in the event of an error. Finally will run whether the error is caught or not

·      Test Driven Development(TDD) – A style of coding where you write the tests before any other code

·      Testing Frameworks provide a wide range of pre-written tests

·      Jest is a framework that on one command (jest -c) will run every file that ends in test.js and neatly display the results

 

Single Page Applications

 

·      A Single Page Application means that clicking an internal link doesn’t cause a server request

·      In theory, once the homepage is loaded you don’t need an established internet connection to navigate through the other pages

·      On an Internally-based SPA, that you can only enter a SPA on the homepage because there is only one url that the server can load

·      On a Location-based SPA, a shared link path will actually take you to the correct page, but this still starts at index and the browser navigates there for you

·      This type of page accepts urls with the properties pathname, hash, search

·      You need to define paths for it to work

·      The History API adds support for page navigation and tracking on a SPA

·      Instead of loading a new document object, it will wipe and replace with the data of the new page.

·      pushState(), replaceState() – methods to load other pages

·      go() – does the same but simple. -1 is last page, 1 is next page

·      We need to set a ‘popstate’ event listener to let the router know that the url has changed

 

The Problem With Single Page Apps

 

·      SPA developers tend to write the entire site’s code on one index.js file

·      To make the page load correctly, you need to replicate a ton of features that browsers will do for you!

·      It is recommended to write separate html files so the browser can actually do its magic

·      You can also put each page in its own labeled folder with an index file so the pathname ends up homepage/sidepage instead of homepage/sidepage.html

·      A single page app might load slightly faster, but at what cost? It takes a lot longer to make and is more prone to issues

 

HTML5 template element

 

·      the <template> element can be used to write html code but not have it called immediately by the browser

·      That code waits for a JavaScript file to fill it with the necessary information and place it on the page
