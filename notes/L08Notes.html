JSON Web Tokens

 

·      Web Tokens are a safe way to transfer information over the internet between two parties.

·      Compact enough to fit in a URL, in POST, or in a HTTP header.

·      The biggest benefit is authentication as to who has access to your JSON information. This will keep the information safe from unauthorized usage.

·      You can also use the token as a way to see who has transferred the information to you.

·      They are formatted like this – Header.Payload.Signature

·      The Header part has two parts – the type of token and the desired hashing algorithm.

 

Header Example:

{

  "alg": "HS256",

  "typ": "JWT"

}

 

·      The Payload can be one of three types: reserved, public, and private claims.

·      Reserved claims are optional and vary by preference. Public claims define characteristics of the transfer, and private claims are custom made for the information being shared.

 

Payload Example:

{

  "sub": "1234567890",

  "name": "John Doe",

  "admin": true

}

 

·      The Signature is the custom string that will authenticate you and is based on whatever hashing method you chose.

 

 

·      JSON Web Tokens are returned when someone logs into user credentials.

·      The authorization info and the key should only be stored in the token’s header.

·      This looks like Authorization: Bearer <token>

·      When the sensitive information is stored in the token, it is considered Stateless, or that the information is not seen once in any part of local or online storage.

 

 

·      JSON Web Tokens are faster, lighter, and more integrated for everyone to use than XML or other formats.

 

Develop, Debug, Learn?

 

·      HTML used to be just fields that would make http requests that would be met with formatted html text in return. People would bag on HTML by saying it wasn’t really a dynamic coding language.

·      With the introduction of JavaScript people started to accessorize their frontend code with new extensions.

·      Back then, the only debugging feature was setting up your own alerts and sometimes that would mess with your browser enough that you would have to reboot.

·      Libraries were built to make things easier and more compatible but that trapped us into relying on heavy code that is way too extensive for what we need.

·      Nowadays, instead of just developing code, we end up picking and choosing other people’s code and depending on them.

·      This style of work really takes away from the debugging because we didn’t write most of the code that is being executed.

·      The focus should be less on us and more on the end users of our sites.

·      We are the delivery drivers of the sites, and the people that use our sites will be on outdated, old, and slow computers. All of the excessively creative features (that we built more for us than for them) will end up slowing or messing with their user experience because they aren’t running the same high-end tech we ran when we tested it.

·      Nobody works on legacy code even when that is still what most people use.

·      Web developers have turned into this elitist group that turns their back on most website users.

·      Nobody can know every single piece that is needed to build a complete website. We just know parts or a little of each. We work on a lot of faith that the packages, frameworks, and libraries are compatible with everything they work with.

·      We even depend on stack overflow code to run when we paste it into our code.

·      We have too many options, and feel like we are expected to know them all.

·      Learning should be a bigger part of the developing and the debugging process.

·      Development environments online do just as much and they compile online so you don’t have to stress yourself and your computer!

·      We can consolidate all of our work environment to avoid context switching.
