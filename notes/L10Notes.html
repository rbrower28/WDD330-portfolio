Functions, Execution Context and the Call Stack

 

·      JavaScript goes line by line, storing new data in memory.

·      This is known as the thread of execution.

·      JS will store an entire function on a single variable name and it will not iterate through the function if it isn’t called.

·      Code on the base level of the file is considered global memory (including all the function variables but not function content).

·      This is also known as global variable environment.

·      Regular JavaScript is Single threaded and Synchronous (one line at a time).

·      When you assign a function’s result to a variable, it will first define it as undefined then perform the function and return the result. This is why variable set functions with no return will technically be undefined.

·      Iteration through the function is the thread of execution.

·      A call stack is when you call functions inside other functions. This is like an array in which it dives through the layers and remembers the layers to come back up in the right order.

·      A call stack is like the movie Inception. You can dive into a dream from another dream, but rules will be applied again and you need to go back out in the same order.

 

Callbacks and Higher Order Functions

 

·      Functions that have no parameters are relatively useless to be used in that format.

·      We can make functions more dynamic by using parameters for different outcomes.

·      Execution context – storing information in memory and executing a function.

·      Inside a function the storage is referred to as local memory.

·      Using higher order functions lets us write even less code.

·      Generalizing our function could turn from this:

 

function copy&add(num)

function copy&subtract(num)

function copy&multiply(num)

function copy&divide(num)

 

·      Into this:

 

Function copy&act(num, action)

 

·      We like writing less code!

·      Using callbacks and higher order functions lets us not only run less code for the same thing but also lets us run several paths of code asynchronously so that one issue with an executable path won’t stop other JS processes from happening.
